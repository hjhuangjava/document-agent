# 方案生成智能体系统设计文档 v3

## 1. 核心设计理念

### 1.1 确定性编排 + 异构节点

工作流拓扑由开发人员在编排画布上确定性地配置，不是 LLM 生成。每个节点是工作流中的一个确定步骤，节点间的连线、条件分支、循环回边都是人为定义的。

节点分两种执行模式：

| 节点类型 | 执行方式 | 谁决定调用什么 | 适用场景 |
|----------|----------|----------------|----------|
| **Agent 节点** | LLM 驱动，模型自主决定调哪些工具、调几次 | LLM | 开放性任务：检索、写作、推理 |
| **Tool 节点** | 确定性地调用一个已注册工具，输入输出固定 | 编排配置 | 确定性任务：格式转换、图表渲染、数据校验 |

这个区分是整个系统的核心。Agent 节点内部是不确定的——LLM 可能调用 0 次工具也可能调用 5 次，可能写一个文件也可能写三个。Tool 节点是确定性的——给定输入必然产生对应输出。编排画布控制的是"谁先谁后、什么条件下走哪条路"，不控制 Agent 节点内部的工具调用序列。

### 1.2 数据流模型

节点间数据通过 AgentState 传递，不走文件系统。VFS 只存储最终产物（图表、完整文档），不用于中间传递。

```
[AgentNode] --写 State--> [AgentState] --读 State--> [下一个节点]
     ↑                                              |
     └── LLM 自主调用 Tool ── Tool 输出回 LLM ──────┘

[ToolNode] --读 State 输入--→ [确定性函数] --写 State 输出--→ [AgentState]
```

## 2. 前端系统设计

技术栈：**Next.js + Tailwind CSS + Shadcn UI + React Flow + TipTap + Vercel AI SDK**

### 2.1 用户方案生成端

1. **需求向导舱**：多步骤表单，输入场景参数，支持克隆历史方案风格。
2. **生成进度看板**：SSE 订阅后端事件。左侧时间轴展示节点执行状态；右侧打字机式方案预览。
3. **方案工作台**：生成完成后 Markdown 可编辑，AI 辅助润色，Diff 视图。
4. **反哺按钮**：用户修改后写入知识库作为 Few-shot 数据。

### 2.2 开发者管理后台

1. **技能仓 (SkillHub)**：管理工具注册表。配置工具元数据（名称、描述、入参 Schema），启用/禁用。**只配元数据，不执行任意代码。**
2. **编排画布 (FlowCanvas)**：
   - 左侧**组件面板 (ComponentPalette)**：按分类展示已注册的 Agent 模板和 Tool，拖拽到画布。
   - 中央**连线画布**：React Flow 画布。节点区分两种外观——Agent 节点显示已绑定的工具列表，Tool 节点显示函数签名。
   - 右侧**属性面板 (PropertyPanel)**：
     - Agent 节点：配置 system prompt、绑定的工具集合、LLM 参数（temperature 等）。
     - Tool 节点：配置输入参数绑定（state 绑定 / 静态值）、输出映射到哪个 state 字段。
   - 连线支持**条件边**（配置 condition 对象）和**回边**（配置 max_retries）。
   - 发布时导出拓扑 JSON 到后端。

## 3. 后端系统设计

### 3.1 AgentState 设计

```python
from typing import Annotated, Literal, Any
from typing_extensions import TypedDict
from langgraph.graph.message import add_messages

class ConsistencyResult(TypedDict):
    status: Literal["pass", "fail"]
    violations: list[str]

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    business_context: dict           # 前端传入的业务参数
    data_query_result: str           # 数据查询结果（Markdown 表格）
    draft_content: str               # 已生成方案的文本内容
    consistency_report: ConsistencyResult  # 校验结果
    vfs_artifacts: list[str]         # VFS 产物路径列表
```

**设计要点**：
- 每个节点的输出有明确的 state 字段承载，不靠文件名猜测传递。
- `consistency_report` 只含业务数据（status + violations），不含流程控制信息（retry_count 不在这里）。
- `vfs_artifacts` 只记录最终产物路径，中间过程的数据走 state。
- 删除 `current_stage`——前端从 SSE 事件中的节点名推断当前阶段。

### 3.2 工具注册表

工具是后端预定义的 Python 函数，通过 `@tool` 注册。管理后台可以启用/禁用已有工具，**不能动态执行任意代码**。

```python
from langchain_core.tools import tool

# --- 数据查询 ---
@tool
def query_business_data(scene_id: str, min_score: int = 60) -> str:
    """查询资产特征库。scene_id 为场景标识，min_score 为特征分值过滤下限。
    返回 Markdown 表格格式的设备特征与联动数据。"""
    result = _fetch_from_db(scene_id, min_score)
    return _format_as_markdown_table(result)

# --- 一致性校验 ---
@tool
def check_consistency(draft_content: str, scene_id: str) -> dict:
    """核对方案原文与业务数据的一致性。draft_content 为方案文本，
    scene_id 用于查回原始数据做交叉比对。返回 {status, violations}。"""
    source_data = _fetch_from_db(scene_id, min_score=0)
    violations = _cross_check(draft_content, source_data)
    return {
        "status": "pass" if not violations else "fail",
        "violations": violations
    }

# --- 图表生成 ---
@tool
def generate_chart(chart_type: str, data_points: list[dict], vfs_session_id: str) -> str:
    """绘制图表并存入 VFS。chart_type 可选 'bar'/'pie'，data_points 格式为
    [{label, value}, ...]。返回 VFS 中的文件路径。"""
    import matplotlib
    matplotlib.use("Agg")
    import matplotlib.pyplot as plt
    import uuid

    vfs = _get_vfs(vfs_session_id)
    plt.figure(figsize=(6, 4))
    x = [d["label"] for d in data_points]
    y = [d["value"] for d in data_points]

    if chart_type == "bar":
        plt.bar(x, y, color="skyblue")
    elif chart_type == "pie":
        plt.pie(y, labels=x, autopct="%1.1f%%")
    else:
        return f"不支持的类型: {chart_type}"

    fid = uuid.uuid4().hex[:8]
    filename = f"charts/chart_{fid}.png"
    path = vfs.root / filename
    path.parent.mkdir(parents=True, exist_ok=True)
    plt.savefig(path, format="png", dpi=150)
    plt.close()
    return str(path)

# --- VFS 读写（仅用于最终产物） ---
@tool
def save_to_vfs(filename: str, content: str, vfs_session_id: str) -> str:
    """将内容保存到 VFS 作为最终产物（完整文档、图表等）。"""
    vfs = _get_vfs(vfs_session_id)
    return vfs.write(filename, content)

# --- 注册表 ---
TOOL_REGISTRY: dict[str, callable] = {
    "query_business_data": query_business_data,
    "check_consistency": check_consistency,
    "generate_chart": generate_chart,
    "save_to_vfs": save_to_vfs,
}
```

**与 v2 的关键区别**：
- `check_consistency` 签名变为接收 `draft_content` 和 `scene_id` 两个参数，不再内部硬编码读 `draft.md`。调用者（Agent 或编排逻辑）负责从 state 取值传入。
- `query_business_data` 和 `check_consistency` 各自查库——这是正确的，因为查询目的不同（前者查特征数据，后者查全量数据做比对），且查询时机和数据范围不同。
- `save_to_vfs` 单独作为一个工具，VFS 读写不混入业务工具。
- 无 `read_section`——中间数据走 state，不需要从文件读回。

### 3.3 节点执行器

两种节点类型，两种执行逻辑。

```python
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI

async def execute_agent_node(
    state: AgentState,
    config: dict,
    system_prompt: str,
    tool_names: list[str],
    llm_params: dict | None = None,
) -> dict:
    """Agent 节点：LLM 自主决策工具调用，循环直到没有工具调用为止。"""

    # 1. 取该节点允许使用的工具子集
    tools = [TOOL_REGISTRY[n] for n in tool_names if n in TOOL_REGISTRY]
    if not tools:
        raise ValueError(f"未找到任何已注册工具: {tool_names}")

    # 2. 构建 LLM（绑定工具）
    params = llm_params or {}
    model = ChatOpenAI(model="gpt-4o", streaming=True, **params)
    model_with_tools = model.bind_tools(tools)

    # 3. 第一轮：LLM 决策
    response = await model_with_tools.ainvoke(
        [("system", system_prompt)] + state["messages"],
        config=config,
    )

    # 4. 如果有工具调用，交给 ToolNode 执行，循环
    tool_node = ToolNode(tools)
    messages = [response]
    while response.tool_calls:
        tool_result = await tool_node.ainvoke(
            {"messages": messages},
            config=config,
        )
        messages.append(tool_result["messages"][-1])
        response = await model_with_tools.ainvoke(
            [("system", system_prompt)] + state["messages"] + messages,
            config=config,
        )
        messages.append(response)

    # 5. 从最终消息中提取业务输出，写入对应 state 字段
    #    由编排层的 node_fn 负责（见 3.5）
    return {"messages": messages}


async def execute_tool_node(
    state: AgentState,
    tool_name: str,
    input_bindings: list[dict],
    output_bindings: list[dict],
) -> dict:
    """Tool 节点：确定性调用，直接执行函数。"""

    tool_fn = TOOL_REGISTRY.get(tool_name)
    if not tool_fn:
        raise ValueError(f"未注册工具: {tool_name}")

    # 1. 解析输入
    resolved_inputs = {}
    for binding in input_bindings:
        resolved_inputs[binding["name"]] = _resolve_value(state, binding)

    # 2. 执行
    result = await tool_fn.ainvoke(resolved_inputs)

    # 3. 写入 state
    state_updates = {}
    for binding in output_bindings:
        # result 可能是 dict 或 str
        value = result.get(binding["output_name"]) if isinstance(result, dict) else result
        _set_state_path(state_updates, binding["state_key"], value)

    return state_updates


def _resolve_value(state: AgentState, binding: dict):
    """根据绑定配置从 state 取值或返回静态值"""
    bind_type = binding.get("bind", {}).get("type", "static")
    if bind_type == "state":
        return _get_state_path(state, binding["bind"]["state_key"])
    else:
        return binding["bind"].get("value", binding.get("default"))

def _get_state_path(state: dict, path: str):
    keys = path.split(".")
    current = state
    for key in keys:
        if isinstance(current, dict) and key in current:
            current = current[key]
        else:
            return None
    return current

def _set_state_path(updates: dict, path: str, value):
    keys = path.split(".")
    current = updates
    for key in keys[:-1]:
        current = current.setdefault(key, {})
    current[keys[-1]] = value
```

### 3.4 VFS 设计

```python
import uuid
import shutil
from pathlib import Path

class VirtualFileSystem:
    """会话级隔离的文件存储，仅用于最终产物（图表、完整文档）"""

    def __init__(self, base_dir: str = "/workspace/vfs"):
        self.session_id = uuid.uuid4().hex[:8]
        self.root = Path(base_dir) / self.session_id
        self.root.mkdir(parents=True, exist_ok=True)
        self._artifacts: list[str] = []

    def write(self, filename: str, content: str | bytes) -> str:
        filepath = self.root / filename
        filepath.parent.mkdir(parents=True, exist_ok=True)
        if isinstance(content, bytes):
            filepath.write_bytes(content)
        else:
            filepath.write_text(content, encoding="utf-8")
        self._artifacts.append(str(filepath))
        return str(filepath)

    def read(self, filename: str) -> str:
        return (self.root / filename).read_text(encoding="utf-8")

    def list_artifacts(self) -> list[str]:
        return list(self._artifacts)

    def cleanup(self):
        shutil.rmtree(self.root, ignore_errors=True)
```

**与 v2 的区别**：VFS 角色缩小为"最终产物存储"，中间数据走 AgentState。VFS 生命周期与 checkpoint 解耦——checkpoint 持久化 state，VFS 独立管理文件。恢复时 state 从 checkpoint 恢复，VFS 文件通过 `vfs_artifacts` 路径列表重建索引即可（文件本身不做 checkpoint，太大且不必要）。

### 3.5 从拓扑 JSON 构建工作流

前端下发的拓扑 JSON 中，每个节点声明自己的 `execution_mode`，后端据此选择执行器。

```python
from langgraph.graph import StateGraph, START, END

def build_workflow(topology: dict) -> StateGraph:
    workflow = StateGraph(AgentState)

    for node_cfg in topology["nodes"]:
        nid = node_cfg["id"]
        mode = node_cfg["execution_mode"]  # "agent" | "tool"

        if mode == "agent":
            # Agent 节点：配置 prompt 和工具列表
            system_prompt = node_cfg["agent_config"]["system_prompt"]
            tool_names = node_cfg["agent_config"]["tool_names"]
            llm_params = node_cfg["agent_config"].get("llm_params")
            output_bindings = node_cfg.get("output_bindings", [])

            async def agent_fn(state, _cfg=node_cfg, _prompt=system_prompt,
                              _tools=tool_names, _llm=llm_params, _out=output_bindings):
                result = await execute_agent_node(state, config={},
                    system_prompt=_prompt, tool_names=_tools, llm_params=_llm)

                # 从对话中提取最终文本，映射到 state
                state_updates = {"messages": result["messages"]}
                final_content = result["messages"][-1].content
                for ob in _out:
                    _set_state_path(state_updates, ob["state_key"], final_content)
                return state_updates

            workflow.add_node(nid, agent_fn)

        elif mode == "tool":
            # Tool 节点：确定性执行
            tool_name = node_cfg["tool_config"]["tool_name"]
            input_bindings = node_cfg["tool_config"]["input_bindings"]
            output_bindings = node_cfg["tool_config"].get("output_bindings", [])

            async def tool_fn(state, _tn=tool_name, _in=input_bindings, _out=output_bindings):
                return await execute_tool_node(state, tool_name=_tn,
                    input_bindings=_in, output_bindings=_out)

            workflow.add_node(nid, tool_fn)

    # 注册边
    _register_edges(workflow, topology["edges"])

    return workflow


def _register_edges(workflow: StateGraph, edges: list[dict]):
    # 分离普通边和条件边
    plain_edges = [e for e in edges if not e.get("condition")]
    cond_edges = [e for e in edges if e.get("condition")]

    # 普通边
    for edge in plain_edges:
        src = START if edge["source"] == "__start__" else edge["source"]
        tgt = END if edge["target"] == "__end__" else edge["target"]
        workflow.add_edge(src, tgt)

    # 条件边：按 source 分组
    groups: dict[str, list] = {}
    for edge in cond_edges:
        groups.setdefault(edge["source"], []).append(edge)

    for src, group in groups.items():
        max_retries = max((e.get("max_retries", 3) for e in group), default=3)
        # 在 state 中维护 retry 计数（由条件路由读写）
        counter_key = f"_retry_count_{src}"

        def make_router(edges_list, _max=max_retries, _ck=counter_key):
            def router(state: AgentState) -> str:
                count = state.get(_ck, 0)
                for e in edges_list:
                    cond = e["condition"]
                    actual = _get_state_path(state, cond["field"])
                    if _eval_condition(actual, cond["operator"], cond["value"]):
                        # 如果是回边（target 在 edges_list 中也是 source），检查计数
                        if e["target"] != "__end__" and count >= _max:
                            return END
                        return e["target"] if e["target"] != "__end__" else END
                return END
            return router

        workflow.add_conditional_edges(src, make_router(group))


def _eval_condition(actual, operator: str, expected) -> bool:
    ops = {
        "eq": lambda a, b: a == b,
        "ne": lambda a, b: a != b,
        "gt": lambda a, b: a is not None and b is not None and a > b,
        "lt": lambda a, b: a is not None and b is not None and a < b,
        "gte": lambda a, b: a is not None and b is not None and a >= b,
        "lte": lambda a, b: a is not None and b is not None and a <= b,
        "in": lambda a, b: a in b if isinstance(b, list) else False,
        "contains": lambda a, b: b in a if isinstance(a, str) else False,
    }
    return ops.get(operator, lambda a, b: False)(actual, expected)
```

### 3.7 Checkpoint 持久化

```python
from langgraph.checkpoint.sqlite.aio import AsyncSqliteSaver
# 生产环境用 PostgresSaver

async def run_workflow(workflow_id: str, session_id: str, business_context: dict):
    topology = await load_topology(workflow_id)
    graph = build_workflow(topology)

    # 编译时注入 checkpointer
    checkpointer = AsyncSqliteSaver.from_conn_string("./checkpoints.db")
    app = graph.compile(checkpointer=checkpointer)

    config = {"configurable": {"thread_id": session_id}}
    inputs = {"business_context": business_context}

    async for event in app.astream_events(inputs, config=config, version="v2"):
        yield event
```

Checkpoint 只持久化 AgentState（内存数据），不持久化 VFS 文件。崩溃恢复后：
1. 从 checkpoint 恢复 AgentState
2. 检查 `vfs_artifacts` 中的路径，文件还在的直接复用（VFS 配置 TTL 延迟清理）
3. 文件丢失的节点重新执行（VFS 产物是可重建的）

### 4.1 Agent 节点

字段说明：


## 5. SSE 事件协议

后端将 LangGraph 内部事件翻译为业务事件，前端只感知业务语义。

```python
async def translate_events(app, inputs, config):
    """翻译层：LangGraph 内部事件 → 业务事件"""
    node_name_map = await load_node_display_names(config)

    async for event in app.astream_events(inputs, config=config, version="v2"):
        kind = event["event"]
        node = event.get("name", "")

        if kind == "on_chain_start" and node in node_name_map:
            yield {
                "event": "node_started",
                "data": {
                    "node_id": node,
                    "node_name": node_name_map[node],
                    "mode": "agent" if _is_agent_node(node, config) else "tool"
                }
            }

        elif kind == "on_chain_end" and node in node_name_map:
            yield {
                "event": "node_completed",
                "data": {
                    "node_id": node,
                    "node_name": node_name_map[node]
                }
            }

        elif kind == "on_tool_start":
            yield {
                "event": "tool_invoked",
                "data": {
                    "tool_name": event["name"],
                    "by_node": _find_parent_node(event)
                }
            }

        elif kind == "on_tool_end":
            yield {
                "event": "tool_result",
                "data": {
                    "tool_name": event["name"],
                    "summary": str(event["data"].get("output", ""))[:200]
                }
            }

        elif kind == "on_chat_model_stream":
            chunk = event["data"].get("chunk")
            if chunk and hasattr(chunk, "content") and chunk.content:
                yield {
                    "event": "text_delta",
                    "data": {"content": chunk.content}
                }
```

前端接收的事件类型：

| 事件 | 含义 | 前端行为 |
|------|------|----------|
| `node_started` | 某节点开始执行 | 画布上该节点高亮/闪烁 |
| `node_completed` | 某节点执行完成 | 节点变绿 |
| `tool_invoked` | Agent 调用了某工具 | 进度看板显示工具调用 |
| `tool_result` | 工具返回结果 | 进度看板显示摘要 |
| `text_delta` | LLM 流式输出文本 | 打字机效果追加到预览区 |
| `approval_required` | HITL 节点等待确认 | 弹出确认对话框 |

## 6. 安全设计

### 6.1 工具注册：配置化，非代码执行

管理后台的操作：
- 启用/禁用已有工具 → 更新数据库 `enabled` 标记
- 修改工具的 `description` → 影响 LLM 的工具选择
- 修改工具的参数默认值 → 影响调用行为

**不允许的操作**：上传任意 Python 代码并执行。

新增工具的流程：
1. 开发者在代码仓库中编写 `@tool` 函数
2. 提交 PR → Code Review → 合并
3. 部署后在管理后台启用

这是唯一安全的路径。动态 `exec()` 用户提交的代码是 RCE 漏洞。

### 6.2 Agent 节点沙箱

Agent 节点的 LLM 只能调用 `tool_names` 中声明的工具子集。LLM 无法调用未绑定工具，无法访问文件系统，无法执行系统命令。

### 6.3 输入校验

所有从前端传入的 `business_context` 在写入 AgentState 前做 Schema 校验（JSON Schema 或 Pydantic model），防止注入。
